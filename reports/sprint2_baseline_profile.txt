Methodology note
-----------------
This document is a *code-anchored* profile, not a live trace. The Anthropic
session that produced these changes cannot reach the docker stack, so a
real flamegraph / OpenTelemetry capture is out of scope. The numbers above
are the only authoritative datapoints. Every hot-path callout below cites
the file and line range; an operator running the live trace can confirm
or refute each hypothesis quickly.

Hot-path budget (likely contributors, ranked by suspected p99 impact)
---------------------------------------------------------------------

#1  JWT validation                                         services/gateway/auth.py:50-90
    Current: every /execute hits Redis (acp:token_validation:{sha256})
    + a SECOND Redis call (acp:token:{hash}) confirming Identity issued
    the token. Two round-trips per request. Under load the Redis client
    pool contention adds 30-150ms to p99 even though individual ops are
    ~1ms.
    Sprint plan: layer an in-process LRU in front of the Redis cache.
    LRU hit returns in <1µs; Redis remains the shared source of truth
    so cross-process invalidation works after logout/rotation.

#2  ResilientClient default timeouts                       sdk/common/resilient_client.py:118-125
    Current: timeout=2.0s with connect=1.0s (= timeout * 0.5).
    When a downstream (behavior, policy, audit) is degraded but not
    yet down, every retry waits the full 2s read budget. With 1 retry,
    a single request can spend 4s in network calls before falling back.
    This is the dominant p99.99 source under brownouts.
    Sprint plan: connect=100ms, read=250ms via settings overrides.
    Plumb settings.RESILIENT_CONNECT_TIMEOUT / RESILIENT_READ_TIMEOUT
    so individual services can override without code changes.

#3  Decision-pipeline gather                               services/decision/main.py:186-198
    Current: asyncio.gather over (policy, behavior) with read=800ms,
    total cap 1.5s (DECISION_GATHER_TOTAL_TIMEOUT). Sprint 1.1 already
    bumped this from 1.0s; it's tight enough already. Further
    parallelism opportunity is limited — the user's task description
    mentions "inference, anomaly, cost, cross_agent" but those are
    *inputs* assembled into the DecisionContext, not separate HTTP
    fan-outs. No additional gather work needed here.

#4  Audit xadd timeout                                     services/gateway/middleware.py:1320-1358
    Current: asyncio.wait_for(log_audit_stream, timeout=1.0). Redis
    xadd is sub-ms when healthy. The 1.0s wait kicks in only on
    pathology (Redis hang) but when it does it drags p99 by exactly
    1s. Lowering to 250ms keeps the audit-on-hot-path contract while
    capping the worst case.

#5  pgbouncer transaction pool                             infra/pgbouncer.ini:23-27
    Current: default_pool_size=25, reserve_pool_size=10. At 200 users
    × ~3 DB-touching services per request, the pool queues briefly
    under bursts. Bump to default_pool_size=50, reserve_pool_size=20.
    pool_mode=transaction is already correct for the read-heavy
    services.

#6  Behavior firewall under load (Sprint 1.1 invariant)    services/decision/behavior_consult.py
    The consult must run on ≥99% of requests. If breaker-skipping
    happens because behavior is brown, classify the consult as
    service_status="skipped" and let degraded_mode_policy decide
    (block_high_risk default). Never silently bypass.

Not in scope this sprint
-------------------------
- Server-side rendering, gzip, HTTP/2 — these don't move the needle on
  a JSON control-plane API of <2KB payloads.
- OPA policy cache — already cached in-process per services/gateway/client.py.
- Receipt signing — happens on /receipts/* endpoints, not /execute.

Expected outcome
----------------
The five hot-path interventions above target compounding ~30-60% latency
reductions each. JWT in-process cache alone is expected to remove 30-150ms
from p99 (the most consistent win). ResilientClient timeout tightening is
the lever for p99.99 — the 4s-on-brownout cliff drops to <500ms.
