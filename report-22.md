### Investigation of the single 200 OK

Claude's mass-assignment attack:

```
POST /demo/spawn-workspace
Authorization: Bearer user_token_abc123          (silently ignored — endpoint is anon)
Content-Type: application/json
{
  "workspace_name":   "test-workspace-34",
  "description":      "Testing mass assignment",
  "role":             "admin",
  "tenant_id":        "malicious-tenant-999",
  "owner_id":         "attacker-user-id"
}
```

Aegis response (200 OK):

```json
{
  "success": true,
  "data": {
    "tenant_id":   "dc07f2e6-9f16-41d4-a989-0bc937e4e053",
    "owner_email": "demo+dc07f2e6@aegisagent.in",
    "jwt":         "eyJhbGciOiJIUzI1NiIsInR5cCI6Ikp..."
  }
}
```

**Why this is NOT a bypass:**
- Server-generated UUID `dc07f2e6-…` — *not* the attacker-supplied `"malicious-tenant-999"`.
- Server-generated email `demo+dc07f2e6@aegisagent.in` — *not* the attacker-supplied `"attacker-user-id"`.
- `role` and `owner_id` body fields silently dropped — server assigned roles based on its own logic.
- This matches the `expected_defence` field Claude itself wrote: *"The platform should reject or ignore role and tenant_id parameters in the request body, assigning them server-side based on authenticated user context only."*

The 200 is the correct success status for a demo-workspace spawn (anonymous public endpoint, by design — Section 1, P1-1 in `22-testing-report.md`). The mass-assignment vulnerability is **closed**.

**Net result: 0 actual bypasses, 0 server errors.**

### Where the "post-WAF-apply FAIL rows" came from (and why they're not a regression)

After today's `terraform apply` landed the WAF `UnAuthPerIPRateLimit` rule (200 unauthenticated requests per 5 minutes per source IP), running the harnesses *back-to-back* burns my own IP's budget. A subsequent anon probe gets `WAF 403` instead of `gateway 401`. The harness scripts mark this as `FAIL` because they assume the response shape (`401 + WWW-Authenticate: Bearer realm="aegis"`) of the gateway, not the WAF.

Demonstration that this is **WAF doing its job**, not a regression:

```
$ curl -o /dev/null -w "%{http_code}\n"  https://aegisagent.in/receipts/key
403                                           ← WAF block (my IP over 200/5min budget)

$ curl -o /dev/null -w "%{http_code}\n" -H "Authorization: Bearer xxx"  https://aegisagent.in/receipts/key
200                                           ← scope_down bypassed → gateway → wrapped envelope JSON

$ curl -o /dev/null -w "%{http_code}\n" -H "Authorization: Bearer xxx"  https://aegisagent.in/workspace/me
401                                           ← gateway answers; auth gate rejects garbage bearer

$ curl -o /dev/null -w "%{http_code}\n" -H "Authorization: Bearer scim_x"  https://aegisagent.in/scim/v2/Users
401                                           ← SCIM router rejects garbage; not 500 (P0-1 closed)
```

So **no auth bypass exists**; the harness just sees the WAF in front of the gateway when its 200-request budget is exhausted. To re-run the harness with clean gateway-401 visibility, wait 5 minutes (WAF rolling window) or fire each probe with `Authorization: Bearer x` (sentinel header to bypass the unauth scope_down).
